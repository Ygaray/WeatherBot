<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| clean SIGTERM vs fatal exit → systemd | The exit code the daemon returns decides whether systemd treats the death as clean (no restart) or a failure (restart→start-limit); a wrong code is a lifecycle-tampering risk. |
| systemd unit directives → OS supervision | Mis-sectioned start-limit directives are silently ignored, defeating the crash-loop bound. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-29-03 | Tampering (lifecycle) | `run_daemon` exit code (fatal vs clean) | high | mitigate | Task 1 `test_fatal_exit_code` + `test_clean_shutdown_returns_zero` pin the separate-marker distinction so a clean SIGTERM never spuriously restarts and a fatal config error always exits non-zero. |
| T-29-04 | Denial of Service (crash-loop) | systemd start-limit placement | medium | mitigate | Task 3 asserts StartLimit* live in `[Unit]` (Pitfall 3) so the crash-loop bound is actually honored, not silently dropped. |
| T-29-05 | Tampering (regression) | AUTH_FAILED classification | high | mitigate | Task 1 `test_auth_not_fatal` guards D-03 so a 401/403 is never turned fatal (which would kill a still-propagating key). |
</threat_model>
