<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| committed pyproject → host deploy | a leaked dev path source would break the host install |
| git tag → resolved sha | a mutable/re-pointed tag could drift the deployed code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-04 | Tampering / DoS | dev path-source leaks into the committed deploy artifact | mitigate | The committed default is the git tag pin (no path source in `[tool.uv.sources]`); the `uv build --no-sources` gate (Task 3) fails the build if a path leaked. Dev co-dev uses the uncommitted venv-overlay (`uv pip install -e ../YahirReusableBot`), never a committed pyproject edit (D-05). |
| T-28-05 | Tampering | mutable git tag re-pointed to a different sha | mitigate | `uv.lock` pins the resolved sha; `uv sync --frozen` reinstalls byte-identically with no re-resolve. Deploy tags treated as immutable (never force-pushed, Pitfall 2). |
| T-28-06 | Tampering | discord.py version drift breaking the live panel | mitigate | The module's EXACT `==2.7.1` pin flows transitively; `uv.lock` is verified to show `2.7.1` (Task 1). The custom_id golden (Task 2 suite) is the oracle. |
| T-28-07 | Spoofing | dependency confusion (slopsquat of `yahir-reusable-bot`) | accept | The dep is a git URL pin, not a registry name — there is no PyPI package to confuse with; `--no-sources` would fail (no published metadata), which is expected and not a deploy path (28-RESEARCH Security Domain). |
</threat_model>
