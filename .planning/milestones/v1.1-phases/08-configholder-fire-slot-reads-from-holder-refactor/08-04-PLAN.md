<objective>
Wire the daemon to read live config from the holder. Refactor `fire_slot` to take a
`holder: ConfigHolder` plus an optional `config: Config | None = None` override (the override
WINS when passed — D-01), reading `holder.current()` ONCE at the top and threading that single
snapshot through the whole delivery (reliability budget read + `send_now(config=snapshot)`) so a
mid-fire `replace()` can never tear one delivery (Pitfall #9). Route `_register_jobs`,
`_run_catchup`, and `_announce_schedule` through the holder; replace the
`add_job(kwargs={"config": config})` capture with `{"holder": holder}`; leave the stable job
`id=f"{name}|{time}|{days}"` and `_heartbeat_tick` UNTOUCHED (D-03). Construct one `ConfigHolder`
in `run_daemon` alongside `stop` and thread it into the three readers (Discretion #4). Update the
single `_register_jobs` positional test callsite to pass a holder. This is the correctness
prerequisite: after it, an UNCHANGED job renders whatever the holder currently owns — so a
Phase 9 reload actually changes what it sends.

Purpose: This closes the loop the whole milestone needs. Plan 03 built the holder; this plan makes
every live job consume `holder.current()` at fire time instead of a baked-in kwarg, then proves it
with the `replace()`-changes-an-unchanged-job test. Output: a refactored `daemon.py` + one updated
test callsite, turning the remaining Plan 01 integration tests GREEN and keeping all 215 green.
</objective>
