<objective>
Add the two operator-facing CLI surfaces: `weatherbot check-config` (CFG-08 offline dry-run) and
`weatherbot reload` (CFG-02 explicit trigger, sender half), plus a NEW `weatherbot/ops/pidfile.py`
ops helper (atomic PID write + read + the `/proc/<pid>/cmdline` staleness guard, D-03).
`check-config` calls the shared `validate_config_and_templates` from Plan 02 — OFFLINE, zero
network, NOT `do_check`/`run_self_check` (Pitfall 8) — and reports pass/fail. `reload` reads the PID
file, verifies the PID is actually a weatherbot process (PID-recycling defense), and sends `SIGHUP`.
Both subcommands slot into the established `add_subparsers` grammar and dispatch ladder.

Purpose: This is the cross-process control path and the operator's pre-flight validation, both
composed over the Plan 02 validator and a small stdlib-only ops helper (mirroring `sdnotify.py`).
The daemon-side SIGHUP receiver + PID write/unlink is Plan 05 (same `daemon.py` file, later wave).
Output: a new `pidfile.py`, the two subparsers + `do_reload` in `cli.py`, and the green
`check-config` exit-matrix + no-network tests plus `test_reload_cli_signals_pid` from Wave 0.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `weatherbot reload` (short-lived process) -> running daemon | A separate UID-local process discovers the daemon PID and sends it a signal — the new cross-process control path; PID recycling could redirect the signal to an unrelated process. |
| operator-edited config -> check-config offline validator | check-config is the operator's pre-flight gate over untrusted hand-edited config; it must validate without ever sending or probing. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-06 | Spoofing / Tampering | SIGHUP delivered to a recycled PID (signal hijack — wrong process killed/disturbed) | mitigate | `is_weatherbot_pid` reads `/proc/<pid>/cmdline` and confirms `b"weatherbot"` BEFORE `os.kill`; a missing/recycled PID returns 1 without signaling (D-03). Proven by `test_reload_cli_signals_pid`. |
| T-09-07 | Tampering | A torn/partial PID file read mid-write | mitigate | `write_pid_atomic` uses temp + `os.replace` (atomic on POSIX) — a reader never observes a partial PID file (Don't Hand-Roll). |
| T-09-04 | Access Control | A non-owner triggering a reload | accept | `os.kill` is OS-enforced to the same UID (V4); single-user personal daemon — no additional auth surface warranted (ASVS L1). |
| T-09-08 | Information Disclosure | check-config / reload logs leaking secrets | mitigate | Outcome-only logging (T-04-01): pass/fail, reason, pid — never a key or webhook URL; check-config never constructs Settings. |
| T-09-SC | Tampering | npm/pip/cargo installs | accept | Zero packages installed; stdlib `os`/`signal`/`tempfile` only; `uv.lock` unchanged. |

Repudiation/DoS/Elevation: N/A — the reload sender is a UID-local signal; no network input, no privilege change.
</threat_model>
