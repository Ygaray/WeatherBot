<objective>
Add the optional stable `Location.id` (D-01) and the single shared offline validator
`validate_config_and_templates` (D-05/D-08) — the two config-layer foundations the whole reload
engine and `check-config` hang on. `id` defaults to the RAW `name` verbatim (Option A from
09-RESEARCH Pitfall 1 — the resolved D-01 contradiction) so the sent-log key is byte-identical for
any config that omits it (zero migration); casefold is used ONLY for the uniqueness check. The
validator wraps the existing `load_config` + `assert_unique_names` (extended for unique id) + the
existing regex `validate_template` (NO Jinja2, zero network) into the ONE function both
`check-config` (CFG-08) and the reload engine (CFG-04) call — so a config that passes one is
exactly a config the other accepts.

Purpose: This is the pure, network-free, fail-loud config layer the rest of Phase 9 composes over.
Output: a refactored `models.py` (the id field + frozen after-validator) and `loader.py` (the
shared validator + unique-id check), turning the Wave-0 `test_models.py` id tests and the shared-
validation contract green. No daemon, CLI, or store change here — those are Plans 03–05.
</objective>
