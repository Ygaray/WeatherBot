# Phase 17 — UI Design Contract

> Interaction and copywriting contract for a **Discord message-component panel**
> (`discord.ui.View` with a location `Select` + command `Button`s). Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Surface note (load-bearing):** This is NOT a web/HTML/CSS frontend. The chrome
> (colors, fonts, pixel spacing, surfaces) is rendered by the Discord client and is
> **not controllable** by this project. The design contract here is therefore a
> **component layout + interaction-state + copywriting** contract scoped to Discord's
> component model and hard caps. Web-centric template sections (Spacing scale,
> Typography, Color palette) are marked **N/A — Discord-rendered; not controllable**
> rather than filled with invented values. The checker should evaluate the
> Discord-appropriate dimensions (Copywriting, Layout, Interaction states,
> Accessibility/clarity) and treat the N/A sections as intentionally not-applicable.
>
> Every decision below is **LOCKED** in `17-CONTEXT.md` (D-06..D-15) and grounded in
> `17-RESEARCH.md` (discord.py 2.7.1, verified against installed source). This file
> consolidates them into a verifiable contract — it does not re-open them.

## Component Layout Contract (replaces "Spacing Scale" for this surface)

> The web "Spacing Scale" section does not apply. The spatial contract for a Discord
> panel is **row/column placement within Discord's hard caps**, which discord.py partly
> enforces and partly does NOT (see the build-time assert below). LOCKED by **D-10**.

**Discord hard caps (the layout budget):** ≤ 5 action rows; ≤ 5 buttons per row; ≤ 25
children total; a `Select` occupies an entire row by itself; `custom_id` ≤ 100 chars;
`label` ≤ 80 chars; `Select` ≤ 25 options.

**Locked layout (D-10):**

| Row | Contents | Component(s) | Count |
|-----|----------|--------------|-------|
| 0 | Location selector (own row) | `Select` (`wb:loc:select`) | 1 select |
| 1 | Location-acting commands | `weather` · `uv` · `next-cloudy` · `sun` · `wind` (5 buttons) | 5 buttons |
| 2 | Argless commands | `status` · `alerts` (2 buttons) | 2 buttons |
| 3–4 | **Reserved — left empty** for Phase 19 (forecast tier) / Phase 20 (polish) | — | 0 |

**Ordering is part of the contract** (D-06 curated, ordered name tuple): row 1 order is
exactly `weather, uv, next-cloudy, sun, wind`; row 2 order is exactly `status, alerts`.
The order is reviewable and deterministic — a button must not silently reorder or appear.

**Build-time layout assertion (LOCKED, D-10 / RESEARCH Pitfall 5) — minimal version now:**
Add an `__init__` guard asserting:

| Bound | Value | Who enforces |
|-------|-------|--------------|
| rows | ≤ 5 | assert here (cheap) + `add_item` ValueError |
| buttons per row | ≤ 5 | `add_item` ValueError |
| total children | ≤ 25 | `add_item` ValueError |
| each `custom_id` length | ≤ 100 | **assert here — discord.py does NOT enforce at construction** (silent until send-time `HTTPException`) |
| each `label` length | ≤ 80 | **assert here — discord.py does NOT enforce at construction** |
| `Select` options | ≤ 25 | `add_option` ValueError |

> Rationale (RESEARCH Pitfall 5, empirically verified): a 101-char `custom_id` and an
> 81-char `label` both construct silently and only fail at Discord send-time with a
> generic error and no panel. The id/label asserts are the ones the project MUST add
> itself. This is the **minimal** assert; Phase 19 lands the full one.

**Persistence precondition (LOCKED, D-10 + RESEARCH Pattern 1):** the assembled view MUST
satisfy `view.is_persistent() == True` (requires `timeout=None` AND every child carries an
explicit static `custom_id`) so Phase 18 can `add_view` it without `ValueError`. Phase 17
does not register it persistently — it only guarantees the construction is registrable.

## Color

**N/A — Discord-rendered components; not controllable.**
Surface/background/accent colors are the Discord theme's. The one color-adjacent lever is
`discord.ButtonStyle` (e.g. `primary`/`secondary`), which is a *style enum*, not a hex token.

**Style contract (the controllable slice):**

| Element | Style | Note |
|---------|-------|------|
| All command buttons (rows 1–2) | a single consistent `ButtonStyle` (e.g. `primary`) | Uniform — buttons are all read-only command triggers of equal weight; do not color-code per command (no semantic meaning is carried by color this phase). |
| Destructive style | **none used** | This phase has **no destructive actions** (panel is read-only — see Copywriting). Do not use `ButtonStyle.danger`. |

**Accessibility constraint (LOCKED, emoji-free phase):** meaning MUST NOT rely on color or
emoji. Every button's purpose is conveyed by its **text label** alone (emoji is Phase 20).
A colorblind or emoji-stripped client must be able to operate the panel from labels only.

## Interaction State Contract (the load-bearing section for this phase)

> This is where Phase 17's correctness lives. All LOCKED in D-11..D-15, verified in RESEARCH
> against discord.py 2.7.1 source.

### State machine per tap

| State | Trigger | Visual | Mechanism | Locked by |
|-------|---------|--------|-----------|-----------|
| **Default** | panel idle | All components enabled; embed shows last result (or nothing on first paint) | assembled `PanelView` | D-10 |
| **Fetching (transient)** | operator taps a fetch-backed button | Panel content shows `⏳ Fetching…`; **all components disabled** (neutralizes double-taps) | the SINGLE `interaction.response.edit_message(content="⏳ Fetching…", view=<disabled copy>)` — this one call simultaneously **acks** (PANEL-05), **paints the cue in-place** (PANEL-06, no new message), and **disables** the panel | D-14 |
| **Result (in-place)** | off-loop fetch returns | Panel re-renders the result embed in place; components **re-enabled** | `await interaction.edit_original_response(content=None, embed=render_embed(reply), view=<re-enabled panel>)` — the followup path, never a 2nd `response.*` | D-14 |
| **Reject (non-operator)** | non-operator taps anything | Foreign user sees an **ephemeral** generic message only; the shared panel is **untouched**; no command runs | `interaction_check` → `send_message(<generic>, ephemeral=True)` then `return False` | D-11/D-12 |
| **Error (in-place)** | callback raises / `UnknownLocationError` | Generic in-place error edit; panel components restored; never a dead button | per-callback `try/except` + `View.on_error` backstop → `_safe_error_edit` (is_done()-guarded) | D-13 + RESEARCH Pattern 4 |

### Hard interaction rules (contract the checker can hold the impl to)

1. **Single-ack rule (D-14/D-15):** exactly **one** `interaction.response.*` call per tap,
   spent on `edit_message`. The result and all error replies land via
   `edit_original_response` / `followup` — **never** a second `response.*` (would raise
   `InteractionResponded`).
2. **No silent defer (D-15):** do **not** use bare `defer()` (a component defer is
   `DEFERRED_UPDATE_MESSAGE` and shows **no** spinner — the panel would sit inert during a
   cold 1–3s fetch and invite re-taps). Do **not** use `defer(thinking=True)` (forces a
   *separate* ephemeral followup, splitting rendering across two surfaces — rejected).
3. **3-second ack guarantee (PANEL-05):** the `edit_message` ack fires **before** the fetch,
   inside Discord's 3s window, so a slow OpenWeather call never shows the client's
   "This interaction failed" toast.
4. **In-place only (PANEL-06):** results render by **editing the panel message** — never a
   new message, modal, or channel send.
5. **Disabled-during-fetch (D-14):** the transient view disables all components so a
   double-tap during the fetch is a no-op. **Acceptable lighter fallback (D-15-blessed):**
   "disable-only, no transient `⏳` text" — keeps the anti-double-tap guarantee if the
   transient-content bookkeeping proves troublesome. Either satisfies the contract; the
   `⏳ Fetching…` text is the preferred form (design-conscious: always give a visible
   "working" cue).
6. **Hot-reload re-derivation (PANEL-02):** the dropdown options are re-derived from
   `holder.current().locations` (lock-free snapshot) so a config hot-reload that adds/removes
   a location is reflected; `config` is read per tap.

## Copywriting Contract

> Labels are **Claude's Discretion** plain text (D-136) within the locked constraints:
> emoji-free, clear, ≤ 80 chars. The wording below is the **prescribed** contract — the
> executor uses these exact strings unless the planner overrides with an equally clear
> emoji-free label.

### Button labels (rows 1–2)

| Command (`custom_id`) | Prescribed label | Takes location? | Source |
|-----------------------|------------------|-----------------|--------|
| `weather` (`wb:cmd:weather`) | `Weather` | yes | D-06/D-07, mirrors `CommandSpec("weather","Weather",…)` |
| `uv` (`wb:cmd:uv`) | `UV` | yes | D-06 |
| `next-cloudy` (`wb:cmd:next-cloudy`) | `Next Cloudy` | yes | D-06 |
| `sun` (`wb:cmd:sun`) | `Sun` | yes | D-06 |
| `wind` (`wb:cmd:wind`) | `Wind` | yes | D-06 |
| `status` (`wb:cmd:status`) | `Status` | **no — `arg=None`** | D-04/D-06/D-09 |
| `alerts` (`wb:cmd:alerts`) | `Alerts` | **no — `arg=None`** | D-04/D-06/D-09 |

**Label rules:** plain text, Title Case, emoji-free, derived from each spec's display name
(keep consistent with `CommandSpec` display names so panel/help/CLI don't drift). No label
may exceed 80 chars (asserted at build time).

### Other copy

| Element | Prescribed copy | Notes / Source |
|---------|-----------------|----------------|
| Dropdown placeholder | `Location` | shown before a selection is visible; the D-03 default already resolves `locations[0]` so a no-selection state is functionally unreachable |
| Transient "working" cue | `⏳ Fetching…` | the only emoji permitted this phase — it is **transient content**, not a button label; the emoji-free rule applies to component labels (D-14). The lighter fallback drops even this. |
| Non-operator reject | `This panel is in use by someone else.` | **LOCKED D-12.** Generic and **identity-free**: NEVER interpolate `interaction.user`, the `custom_id`, the command name, or the operator's identity. Reads as an intentional access decision, not a bug. |
| Error / unknown-location (in-place) | A **generic** helpful message, e.g. the `UnknownLocationError` message string, with **no secret** (token/webhook never appears) | RESEARCH Pattern 2 + Security V7; mirrors `on_message`'s call-site catch. Keep it generic — do not echo internals or stack detail. |
| Empty state | **None required** | The D-03 default selection makes a no-data/no-location empty state unreachable; greying-out-until-selected (PANEL-V2-01) is deferred. No empty-state copy is owned by this phase. |
| Destructive confirmation | **None — no destructive actions** | The panel is strictly read-only (no config edits, no deletes, no sends). No confirmation flow exists or is needed. |

### Copywriting invariants (checker-verifiable)

- Reject copy is **byte-for-byte** `This panel is in use by someone else.` and contains **no**
  interpolated identity (D-12).
- No button label or user-facing string contains an emoji (emoji is Phase 20) — the lone
  exception is the transient `⏳ Fetching…` content string (D-14).
- No user-facing string or log line leaks the OpenWeather appid or Discord webhook/token
  (Security V6/V7; reuse the v1.1 secret-hygiene rule).
- The rejected-attempt **log** is server-side only (`structlog` info/warning with
  user_id / custom_id) and is **never echoed** to the foreign user — and it is the **sole**
  audit record of a rejection, because a clean `interaction_check` `return False` does **not**
  fire `on_error` (D-13, verified in RESEARCH).

## Accessibility / Clarity (Discord-appropriate)

| Concern | Contract | Locked by |
|---------|----------|-----------|
| No color-only meaning | Every command is identifiable by its text label alone; buttons share one uniform style | emoji-free phase, D-10 |
| No emoji-only meaning | Labels are plain text; the only emoji is the transient cue, which is not load-bearing for identifying a control | D-14/D-136 |
| Predictable ordering | Fixed, reviewable order per row (D-06 curated tuple) — controls never reflow | D-06/D-10 |
| Clear placeholder | `Location` placeholder names the selector's purpose | D-03 |
| Slow-fetch never looks broken | The 3s `edit_message` ack + `⏳ Fetching…` cue guarantees feedback before any network latency | D-14/D-15, PANEL-05 |
| Foreign-user clarity | Ephemeral generic reject is visible to the tapper only; no confusing "interaction failed" toast | D-11 |
