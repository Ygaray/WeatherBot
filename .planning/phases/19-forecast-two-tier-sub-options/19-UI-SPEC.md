# Phase 19 — UI Design Contract

> Interaction and copywriting contract for the **Forecast two-tier reveal/collapse**
> work on the existing Discord control panel (`discord.ui.View` + embed). Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Surface note (load-bearing):** This is NOT a web/HTML/CSS frontend. The chrome
> (colors, fonts, pixel spacing, surfaces) is rendered by the Discord client and is
> **not controllable** by this project. The design contract here is a **component
> layout + reveal/collapse interaction + copywriting** contract scoped to Discord's
> component model and hard caps. Web-centric template sections (Spacing scale,
> Typography, Color palette) are marked **N/A — Discord-rendered; not controllable**
> rather than filled with invented values. The checker should evaluate the
> Discord-appropriate dimensions (Copywriting, Layout, Interaction states,
> Accessibility/clarity) and treat the N/A sections as intentionally not-applicable.
>
> **What this phase adds (the whole contract in one breath):** a **Forecast toggle**
> button in row 2 (next to `Status`/`Alerts`) that **reveals** a 2×2 forecast sub-grid
> in rows 3–4, and a **collapse-on-any-action** disclosure model. The four sub-buttons
> (`Weekday Detailed` · `Weekday Compact` / `Weekend Detailed` · `Weekend Compact`) each
> route through the **same** Phase-16 `dispatch_spec` seam the text command uses — no
> parallel forecast logic. The revealed panel hits **5/5 rows, 13/25 children** (full
> height), so the build-time layout assertion is now **load-bearing** and completed here.
>
> Every decision below is **LOCKED** in `19-CONTEXT.md` (D-01..D-09) and grounded in
> `19-RESEARCH.md` (discord.py 2.7.1, verified against installed source). This file
> consolidates them into a verifiable contract — it does not re-open them. The panel's
> existing visual/interaction baseline is **inherited unchanged** from the approved
> `17-UI-SPEC.md` and `18-UI-SPEC.md`.

## Component Layout Contract (replaces "Spacing Scale" for this surface)

> The web "Spacing Scale" section does not apply. The spatial contract for a Discord
> panel is **row/column placement within Discord's hard caps**. LOCKED by **D-06/D-07/D-08**.

**Discord hard caps (the layout budget):** ≤ 5 action rows; ≤ 5 buttons per row; ≤ 25
children total; a `Select` occupies an entire row by itself; `custom_id` ≤ 100 chars;
`label` ≤ 80 chars; `Select` ≤ 25 options.

### Base (collapsed) layout — what the panel shows at rest

| Row | Contents | Component(s) | Count |
|-----|----------|--------------|-------|
| 0 | Location selector (own row) | `Select` (`wb:loc:select`) | 1 select |
| 1 | Location-acting commands | `Weather` · `UV` · `Next Cloudy` · `Sun` · `Wind` | 5 buttons |
| 2 | Argless commands + Forecast toggle | `Status` · `Alerts` · **`Forecast`** | 3 buttons |
| 3–4 | **hidden** in the collapsed render | — | 0 shown |

### Expanded (revealed) layout — after a Forecast tap

| Row | Contents | Component(s) | Count |
|-----|----------|--------------|-------|
| 0 | Location selector | `Select` (`wb:loc:select`) | 1 select |
| 1 | Location-acting commands | `Weather` · `UV` · `Next Cloudy` · `Sun` · `Wind` | 5 buttons |
| 2 | Argless + Forecast toggle | `Status` · `Alerts` · `Forecast` | 3 buttons |
| 3 | **Weekday forecast pair** | `Weekday Detailed` · `Weekday Compact` | 2 buttons |
| 4 | **Weekend forecast pair** | `Weekend Detailed` · `Weekend Compact` | 2 buttons |

**Revealed totals (LOCKED, D-06):** **5/5 rows, 13/25 children** — full row height, **zero
spare row**. Any future component row trips the build-time guard below.

**Ordering is part of the contract** (D-06, curated grid): row 2 is exactly
`Status, Alerts, Forecast` (Forecast last); row 3 is exactly `Weekday Detailed, Weekday
Compact`; row 4 is exactly `Weekend Detailed, Weekend Compact`. The weekday-row / weekend-row
grouping (and the explicit `Detailed`/`Compact` labels over bare `Weekday`/`Weekend`) was
chosen from rendered mockups for scannability and zero ambiguity. Controls never silently
reorder or appear.

**Registration vs. display (LOCKED, D-05 / RESEARCH Pattern 1 — the load-bearing layout
rule):** `PanelView.__init__` builds **all 13 children once** (base + toggle + 4 sub-buttons),
so `add_view(PanelView(...))` in `setup_hook` registers **every** forecast `custom_id` into
the global `(type, custom_id)` dispatch table. Reveal/collapse is a **cosmetic**
`edit_message(view=<freshly-built render view attaching the base-vs-expanded subset>)` — it
**MUST NOT** `add_item`/`remove_item` on the registered persistent view. Routing is
display-independent: a revealed sub-button tapped after a restart still routes because its
`custom_id` is registered, regardless of what the post-restart message currently displays
(this is the D-05 guarantee). Build a fresh render view (mirroring the existing
`_disabled_copy` clone technique); never mutate the registered one.

### Build-time layout assertion (LOCKED, D-08 — COMPLETED this phase)

The panel is now at 5/5 rows, so `_assert_layout` (in `__init__`) becomes load-bearing and is
completed to validate the **full / revealed** layout. Caps split into "discord.py raises" vs
"discord.py accepts silently — we MUST assert":

| Bound | Value | Who enforces | Status entering Phase 19 |
|-------|-------|--------------|--------------------------|
| rows | ≤ 5 | assert here + `add_item` ValueError | present (`_MAX_ROWS`) |
| **buttons per row** | ≤ 5 | **assert here explicitly** (D-08) — currently only `add_item` raised | **ADD** (`_MAX_PER_ROW = 5`) |
| **total children** | ≤ 25 | **assert here explicitly** (D-08) — currently unchecked | **ADD** (`_MAX_CHILDREN = 25`) |
| each `custom_id` length | ≤ 100 | **assert here** — discord.py does NOT enforce at construction (silent until send-time `HTTPException`) | present (`_MAX_CUSTOM_ID`) |
| each `label` length | ≤ 80 | **assert here** — discord.py does NOT enforce at construction | present (`_MAX_LABEL`) |
| `Select` options | ≤ 25 | `add_option` ValueError + assert | present (`_MAX_OPTIONS`) |

**Dedicated test obligation (D-08, criterion 3):** a unit test MUST assert (a) `_assert_layout`
**passes** on the assembled full panel at exactly 13 children / 5 rows / ≤5 per row, AND (b)
an over-cap layout (6th row / 26th child / 6-per-row / 101-char `custom_id` / 81-char `label`)
**trips** the assert — so "a future addition can't silently overflow" is enforced in CI as
well as at construction.

**Persistence precondition (inherited, LOCKED):** the assembled full-child view MUST satisfy
`view.is_persistent() == True` (`timeout=None` + every child a static `custom_id`) so
`add_view` registers it without `ValueError`. All four `wb:fc:*` ids and `wb:forecast:toggle`
are < 100 chars; all four sub-button labels < 80 chars — the id/label asserts pass.

## Interaction State Contract (the load-bearing section for this phase)

> This is where Phase 19's correctness lives: the reveal/collapse disclosure model and the
> single-ack `on_forecast` path. All LOCKED in D-01..D-05/D-09, verified in RESEARCH against
> discord.py 2.7.1 source. The base per-tap fetch state machine (Default / Fetching / Result /
> Reject / Error) is **inherited unchanged** from 17-UI-SPEC and is not re-specified — the new
> paths mirror it.

### Reveal / collapse disclosure model

| State | Trigger | Visual | Mechanism | Locked by |
|-------|---------|--------|-----------|-----------|
| **Collapsed (default)** | panel idle; first paint; any non-toggle action; after a forecast result; after restart | Rows 0–2 only (Select + command grid + Status/Alerts/Forecast). Rows 3–4 hidden. | base render view (expanded=False) | D-03/D-04/D-05 |
| **Expanded (revealed)** | operator taps **Forecast** while collapsed | Rows 0–4 shown; the 2×2 grid appears below | single `interaction.response.edit_message(view=<expanded render view>)` — one ack, cosmetic swap | D-03/D-07 |
| **Re-collapse via toggle** | operator taps **Forecast** while expanded | Returns to collapsed (rows 3–4 hidden) — a plain toggle | single `edit_message(view=<collapsed render view>)` | D-04 |
| **Collapse-on-action** | operator taps any other command button, OR changes the dropdown (`on_select`), OR taps a forecast variant | Renders the **collapsed** base (rows 3–4 hidden) as part of that action's result/render | each callback's terminal render attaches the collapsed view | D-04 |
| **Result-then-collapse** | operator taps a forecast variant | The forecast result renders in place AND the panel returns to collapsed | `on_forecast` result render uses the collapsed view (see below) | D-03 |
| **Post-restart display drift (accepted)** | restart while a panel was last edited expanded | The message may still *display* the sub-grid until the next interaction re-renders it collapsed | Discord persists last-edited components; sub-button `custom_id`s are registered so taps still route, and the next action collapses | D-05 |

### `on_forecast` single-ack path (mirrors `on_command` exactly)

A forecast variant tap follows the inherited single-`response.*`-per-tap contract, differing
only in that it (a) builds `ForecastFlags` directly and passes `flags=`, and (b) collapses on
its result render (D-03):

| Step | Action | Mechanism |
|------|--------|-----------|
| ① Ack + disable (transient) | Show `⏳ Fetching…`; disable all children (neutralize double-taps) | the SINGLE `interaction.response.edit_message(content=_FETCHING_CUE, view=<disabled copy>)` — acks <3s, paints the cue in-place, disables |
| ② Build flags | `ForecastFlags(variant=<"detailed"\|"compact">, location=self._selected_location)` with `add`/`drop` **empty** (the command name encodes the day set) | reads the in-memory `_selected_location` (never re-reads the `Select` — RESEARCH Pitfall 5) |
| ③ Dispatch | `await dispatch_spec(spec, None, …, flags=flags)` off-loop — `spec` is `registry.BY_NAME["weekday-forecast"\|"weekend-forecast"]` | the **same** shared seam the text command uses; `flags is not None` skips `parse_forecast_flags` (D-01) |
| ④ Result + collapse | Re-render the result embed in place AND attach the **collapsed** base view | `await interaction.edit_original_response(content=None, embed=render_embed(reply), view=<collapsed base>)` — the followup path, never a 2nd `response.*` |

### Hard interaction rules (contract the checker can hold the impl to)

1. **One canonical persistent view (D-05/Pattern 1):** the view registered via `add_view` holds
   ALL 13 children. Reveal/collapse renders a **fresh** view attaching a subset — it never
   mutates the registered view. Registering a base-only view, or `remove_item`-ing the
   sub-grid, drops those `custom_id`s from the dispatch table and breaks post-restart routing.
2. **Single-ack rule (inherited D-14/D-15):** exactly **one** `interaction.response.*` per tap,
   spent on `edit_message`. Results and errors land via `edit_original_response` — **never** a
   second `response.*` (would raise `InteractionResponded`). Applies to `on_forecast` and the
   toggle alike.
3. **Collapse on any non-toggle action (D-04):** only the Forecast toggle yields the expanded
   render. A forecast variant, any other command button, and a dropdown change all render the
   collapsed base. This keeps reveal state transient and the model simple.
4. **Result-then-collapse (D-03):** a forecast variant renders its result AND collapses in the
   same terminal `edit_original_response` — the collapse is the result render attaching the
   base view, not a second call.
5. **In-place only (inherited PANEL-06):** forecast results render by editing the panel message
   — never a new message, modal, or channel send.
6. **Disabled-during-fetch (inherited D-09/IN-03):** the transient view disables all children.
   If the toggle and the four variant buttons are `discord.ui.Button` subclasses, the existing
   `_disabled_copy` `isinstance(child, discord.ui.Button)` branch rebuilds them with **no new
   branch** — the planner must **verify** this holds (and prefer merging `_disabled_copy` into a
   single parameterized `_render_view(expanded, disabled)` to kill the IN-03 two-path drift
   risk). If a sub-button is NOT a plain `Button` subclass, a new `_disabled_copy` branch is
   required.
7. **Operator gate covers the new children (inherited D-11/V4):** `interaction_check` runs before
   the toggle and every sub-button callback; a non-operator tap is rejected leak-free exactly as
   for command buttons — no new bypass.
8. **No on-loop I/O added (PANEL-10):** `on_forecast` adds no blocking work on the event loop;
   all fetch/render is inside `dispatch_spec`'s `run_in_executor`.
