<objective>
Two coupled send-path corrections. DELIV-03 (highest-subtlety): fire_slot's `_attempt` (daemon.py:247-259) calls send_now, and send_now (cli.py:142) does BOTH the fetch (lookup_weather at cli.py:196) and the deliver (channel.send_briefing at cli.py:208) in one call — so `retrying(_attempt)` (daemon.py:262) re-runs lookup_weather (a fresh dual OpenWeather fetch) on every retry. The fetch must happen ONCE; only the delivery retries against the in-memory payload. The tension: cli.py:211-216 deliberately keeps the fetch inside the retry so a fetch-429 httpx.HTTPStatusError (Retry-After) reaches the two-burst wait callable (RELY-02) — the hoist must not regress that. DELIV-04: discord.py _post (:106-115) maps ALL non-2xx (incl. 401/403) to DeliveryResult(ok=False) and never raises, so a revoked webhook burns the full ~65-min schedule and records transient_exhausted instead of auth_failed.

Purpose: Correct OpenWeather call volume under retry (no re-fetch on delivery-only failure) and stop mis-classifying a permanent auth failure as transient — reusing the Phase-30 httpx.HTTPStatusError contract so the daemon needs ZERO new classification code.
Output: A fetch-once / deliver-retry restructure (a checkpoint on fetch-429 disposition), the app-side 401/403 raise carrier in discord._post with a redacted URL, and retry-reuse + auth-short-circuit regression tests.
</objective>

| Boundary | Description |
|----------|-------------|
| Discord webhook response → _post → DeliveryResult/exception | The webhook URL is a secret-adjacent token; the auth-failure carrier must not leak it into an exception message, alert detail, or traceback. |
| _post raised exception → fire_slot classifier | The synthesized httpx.HTTPStatusError crosses into is_auth_failure; only .response.status_code is trusted/read. |
| OpenWeather fetch → retried delivery unit | The fetched payload is reused across delivery retries; a re-fetch would change content mid-retry and inflate call volume. |
