## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| config.toml → Schedule model | Hand-edited `time`/`days`/`enabled` tokens cross into the app (untrusted text) |
| app → data/weatherbot.db (sent_log) | Location name + send-time string written/read as SQL parameters |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `sent_log` INSERT/SELECT in store.py | mitigate | Parameterized `?` only (mirror `persist`); `location_name`/`send_time` never f-string'd into SQL; `UNIQUE` constraint backstops races |
| T-03-02 | Tampering | `Schedule.days`/`time` config input | mitigate | pydantic `field_validator` whitelists the `days` vocabulary (`parse_days`) and validates `HH:MM` at load — fail-loud, no bad token reaches the trigger (V5 Input Validation) |
| T-03-03 | Denial of Service | absurd schedule (e.g. every-minute) | accept | Single-user tool; OpenWeather 60/min, 1M/month headroom; 90-min/today catch-up bound (Plan 03) caps bursts — no control needed for v1 |
| T-03-SC | Tampering | apscheduler/time-machine install | mitigate | Task 1 blocking-human legitimacy checkpoint (PyPI verify) before `uv add`; both packages CLAUDE.md/STACK.md-locked |
</threat_model>
