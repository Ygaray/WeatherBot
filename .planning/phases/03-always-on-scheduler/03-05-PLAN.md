<objective>
Close gap #2 from 03-VERIFICATION.md: `fire_slot` performs a `was_sent` READ
(daemon.py:96), then a network + Discord DELIVERY via `send_now` (daemon.py:111), then a
`record_sent` WRITE (daemon.py:123) across separate connections — a non-atomic
check-then-act with the side-effecting delivery INSIDE the window. The `UNIQUE`
constraint dedups ROWS, not DELIVERIES: two overlapping fires for the same slot (catch-up
overlapping the live cron at the same minute, a coalesced misfire, or a fall-back-repeated
minute on the BackgroundScheduler thread pool) both pass the read and both POST — the user
gets two messages. No atomic `claim_slot`/`rowcount` arbitration exists (grep found none).

Root cause: at-most-once ROWS is not exactly-once DELIVERY. The fix prescribed by
03-VERIFICATION.md gap #2 `missing:` and 03-REVIEW.md CR-02: add an atomic
`claim_slot(...) -> bool` (INSERT OR IGNORE + `cur.rowcount == 1` ⇒ this caller won) that
GATES delivery BEFORE the network send; on delivery failure, DELETE the claim so the slot
stays re-fireable (preserve SCHD-06 "always send late" and D-07 mark-after-success intent
for the failure case).

Purpose: restore SCHD-07 / success criterion #5 / the goal's "never sending a slot twice"
at the DELIVERY boundary, not just the row boundary.

Output: an atomic `claim_slot` (+ claim-release) in store.py, a `fire_slot` that claims
before sending and releases on failure, and a concurrent-double-fire test asserting
exactly one POST.
</objective>
