<objective>
Close gap #1 from 03-VERIFICATION.md: `plan_catchup` mis-computes the intended fire
instant across DST transitions, so the missed-send planner silently DISAGREES with the
live APScheduler `CronTrigger` precisely on the days DST handling is supposed to protect.

Root cause (catchup.py:137): `scheduled = now_local.replace(hour=hh, minute=mm, ...)`
keeps `now_local`'s already-resolved UTC offset/fold and does NOT re-resolve it for the
new wall-clock time. Two observed failures:
- Spring-forward gap: a 02:30 slot on 2026-03-08 (02:00→03:00 skipped) is a wall-clock
  time that never occurs, yet `plan_catchup` returns one phantom MissedSlot the live
  trigger would never fire.
- Fall-back fold: a 01:30 slot on 2026-11-01 yields a negative-day delta (`scheduled >
  now_local` true), so the slot is treated as "not due yet" and never caught up.

Purpose: restore SCHD-04's "survives DST" half and success criterion #3 ("no skipped
spring-forward miss, no doubled fall-back send"). The shipping `test_dst_exactly_once`
only exercises a 07:00 slot (far outside the 01:00–02:59 band), so the 128-test green
suite is false confidence (WR-01) — this plan adds transition-band tests that FAIL
against the current code, then makes them pass.

Output: a DST-correct `plan_catchup` and transition-band tests locking the behavior.
</objective>
