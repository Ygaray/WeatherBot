# Phase 18 — UI Design Contract

> Interaction and copywriting contract for the **restart-durability + `!panel` lifecycle**
> work on the existing Discord control panel (`discord.ui.View` + embed). Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Surface note (load-bearing):** This is NOT a web/HTML/CSS frontend. The chrome
> (colors, fonts, pixel spacing, surfaces) is rendered by the Discord client and is
> **not controllable** by this project. Web-centric template sections (Spacing scale,
> Typography, Color palette) are marked **N/A — Discord-rendered; not controllable**
> rather than filled with invented values.
>
> **Scope note (the most important thing about this file):** Phase 18 is a **backend
> restart-durability + lifecycle phase**. It introduces **NO new visual components** and
> **does not change the panel's appearance**. The panel itself — its embed, button grid,
> location dropdown, layout, interaction-state machine, and all in-panel copy — was
> designed and LOCKED in **Phase 17** (`17-UI-SPEC.md`, status: approved). That contract
> is the **locked baseline** for this phase and is referenced, not re-specified, below.
>
> The **only** new user-observable contracts this phase owns are:
> 1. **Operator-facing feedback copy** for the `!panel` summon command (success
>    confirmation, missing-permission refusal, bad/unconfigured-channel abort) — a
>    text-channel message surface, NOT a component.
> 2. The **post-restart selected-location state** (default-on-restart = `locations[0]`),
>    which is already satisfied by Phase 17's in-memory default and adds no new visual.
>
> Every decision is grounded in `18-CONTEXT.md` (D-01..D-13, all LOCKED) and
> `18-RESEARCH.md` (discord.py 2.7.1, verified against installed source).

## Interaction State Contract — `!panel` lifecycle (the new surface this phase owns)

> Phase 18's only NEW interaction is the operator typing `!panel`. It is a **lifecycle
> WRITE command** (D-07) handled in the operator-gated `on_message` path — NOT a component
> click, NOT routed through `dispatch_spec`. The states below are what the **operator** (the
> only person who can trigger it) sees in the text channel. All LOCKED in D-01..D-13.

### `!panel` summon state machine

| State | Trigger | Operator-visible outcome | Mechanism | Locked by |
|-------|---------|--------------------------|-----------|-----------|
| **Summon → reuse** | `!panel`, ≥1 valid panel already pinned | The existing pinned panel is edited in place (keeps its pin position + history); extras deleted; exactly one panel remains. A short confirmation message is sent. | scan `channel.pins()` → `message.edit(embed=…, view=…)` on first match, `message.delete()` on the rest | D-01/D-05/D-06 |
| **Summon → create** | `!panel`, no valid panel found | A fresh panel is posted and pinned; confirmation sent. | post `PanelView` message → `message.pin()` | D-01/D-06 |
| **Refuse → missing permission** | `!panel`, bot lacks any required channel permission | **No panel is posted or edited** (no half-broken/orphan state); operator gets an ephemeral message **naming the missing permission(s)**; a CRITICAL is logged server-side. | eager `channel.permissions_for(guild.me)` preflight → CRITICAL log + ephemeral + REFUSE | D-09/D-10/D-11 |
| **Refuse → revoked mid-write (TOCTOU)** | permission revoked between preflight and a write | The write is caught (`discord.Forbidden`), a CRITICAL is logged; no traceback bubbles; bot thread survives. | per-write `try/except discord.Forbidden` → CRITICAL | D-09 |
| **Abort → channel unconfigured/inaccessible** | `panel_channel_id` unset or points to a deleted/inaccessible channel | Operator gets a clear message that the panel channel is not configured/inaccessible; **bot thread does NOT crash**; nothing posted. | `guild.get_channel(panel_channel_id) is None` → clear log + message + abort | D-04 |
| **Post-restart (passive)** | `systemctl restart weatherbot` (no `!panel`) | The already-pinned panel's buttons + dropdown still route to callbacks (no "interaction failed"); selected location resets to the documented default. | `add_view(PanelView(...))` in `setup_hook` re-binds by `custom_id`; fresh view defaults to `locations[0]` | D-08/D-12/D-13, PANEL-09 |

### Hard interaction rules (checker-verifiable, new to this phase)

1. **No partial/orphan state on a permission failure (SC#4).** The eager preflight runs
   **before** any post/pin; a missing permission means **nothing is written**. A
   posted-but-unpinned orphan is a contract violation. (D-09)
2. **Exactly one panel after summon (SC#2).** Reuse-in-place + delete-extras — never
   unpin-only (an unpinned-but-live view still responds to clicks). (D-06)
3. **Marker-strict ownership.** Only messages matching `author == bot.user` **AND** a
   `wb:`-prefixed component `custom_id` are touched/deleted — never author-alone (must
   never delete an unrelated bot pin). (D-05)
4. **Default-on-restart selected location (SC#3).** After a restart the panel resolves to
   `locations[0]` (mirrors `resolve_location(config, None)`); the next tap hits a valid
   location, never an error. Persisting the prior selection is Out of Scope. (D-08)
5. **Fail loud, not silently dead.** A missing permission, a `Forbidden`, or a bad channel
   each produce a **clear CRITICAL/operator message** — never a silent no-op mid-operation.
   (D-04/D-09/D-11)
6. **No secret leakage.** No operator-facing message or log line contains the bot token or
   OpenWeather appid; `channel_id` (non-secret) may appear in structured logs only. (inherited
   secret-hygiene rule; Security V7)

## Copywriting Contract

> The `!panel` operator-feedback strings are **Claude's Discretion** (18-CONTEXT.md → "exact
> CRITICAL/ephemeral copy strings"). The wording below is the **prescribed** contract — the
> executor uses these exact strings unless the planner substitutes an equally clear, specific,
> emoji-free, secret-free alternative. All inherit Phase 17's emoji-free + identity-free +
> no-secret invariants.

### New operator-feedback copy (the `!panel` text-channel surface)

| Element | Prescribed copy | Notes / Source |
|---------|-----------------|----------------|
| Summon success — reused | `Panel ready — reusing the existing pinned panel.` | Confirms idempotent reuse (D-06); reassures the operator no duplicate was created. Plain text, no emoji. |
| Summon success — created | `Panel ready — posted and pinned a new control panel.` | Confirms the create branch (D-06). |
| Summon success — strays cleaned | `Panel ready — kept one panel and removed {n} stray panel(s).` | Optional, only when extras were deleted; `{n}` is a non-secret count, safe to interpolate (D-06, SC#2). If kept minimal, fold into the reuse message. |
| Missing-permission refusal (ephemeral) | `Can't summon the panel — I'm missing the {perm} permission in that channel.` | **D-11.** Names the **specific** missing permission(s) so the operator can fix it; `{perm}` is the human-readable name(s) of the missing D-10 perm(s) (`view_channel`, `send_messages`, `embed_links`, `read_message_history`, `pin_messages`). No token/id, no stack detail. |
| TOCTOU `Forbidden` (operator-facing, optional) | `Can't summon the panel — a permission was revoked mid-operation.` | The CRITICAL log is the primary record (D-09); an operator message is optional but should stay generic and secret-free. |
| Channel unconfigured/inaccessible | `Panel channel is not configured or is inaccessible — set [bot] panel_channel_id and restart.` | **D-04.** Names the exact config key + the restart requirement (read-once-at-startup boundary). Actionable next step, no secret. |
| CRITICAL log line (server-side) | Structured `structlog` CRITICAL naming the missing permission(s) + `channel_id`, mirroring the existing `on_ready` missing-intent CRITICAL precedent | **D-11.** Log only; never echoed verbatim to the operator beyond the ephemeral above. Non-secret `channel_id` allowed; token/appid never. |

### Inherited / unchanged copy (NOT re-owned here)

| Element | Status |
|---------|--------|
| Button labels, dropdown placeholder, non-operator reject, in-panel error/fetching cue | **Unchanged — owned by Phase 17.** Do not modify. |
| Empty state | **None required.** The `locations[0]` default (D-08) keeps a no-location state unreachable; greying-out-until-selected (PANEL-V2-01) is deferred. |
| Destructive confirmation | **None — `!panel` is not destructive to user data.** It deletes only the bot's **own stray duplicate panels** (D-05/D-06), which is the idempotency mechanism, not a user-facing destructive action. No confirmation flow is added; the operator gate (D-07) is the only access control. |

### Copywriting invariants (checker-verifiable)

- All new operator-feedback strings are **plain text, emoji-free** (emoji is Phase 20).
- The missing-permission message **names the specific permission(s)** — it is not a generic
  "missing permissions" stub (D-11 requires naming the gap so the operator can act).
- The channel-misconfig message **names `[bot] panel_channel_id`** and the **restart**
  requirement (D-04) — actionable, not just "something's wrong."
- No new string interpolates an identity (user/operator) or leaks a secret (token/appid);
  only non-secret values (permission names, stray count, `channel_id` in logs) may appear.
- `!panel` reuses the existing operator gate (D-07): a non-operator never sees ANY of this
  copy — they fall through the `author.id != operator_id` drop with no echo.
