<objective>
Migrate the exactly-once idempotency key's first component from the mutable display `location.name`
to the stable `location.id` (Plan 02) at ALL FIVE callsites IN LOCKSTEP — the four daemon
`fire_slot` store calls (`claim_slot`, `release_claim`, `record_alert`, `resolve_alert`) and the
catchup `was_sent` read. Because `location.id` defaults to the RAW name (Plan 02, Option A), the
id-keyed write/read is byte-identical to existing rows → ZERO data migration. The DB column stays
`location_name` (value changes, no `_SCHEMA` edit — Pitfall 3); `weather/store.py` is untouched.
Human-readable `_log.info(location=location.name)` display fields stay `name`; only the store KEY
argument moves to `id`.

Purpose: This is the structural foundation of the SC#4 / Pitfall #8 exactly-once guarantee — once
the key is keyed on a STABLE id, a reload that changes a slot's tz/name/send_time (keeping the id)
cannot reset its "already sent today" state, so the next fire's `claim_slot` loses and the slot
never re-fires. Doing all five in one change prevents the orphaned-claim / duplicate-alert desync
(Pitfall 2). Output: the keyed daemon + catchup callsites, turning the id-key half of the SC#4
exactly-once test green; the reload engine itself (Plan 05) then proves no re-fire end-to-end.
</objective>

| reloaded config -> sent-log exactly-once key | A reload can change a location's display name/tz/send_time; the key must stay anchored to a STABLE id so a reload cannot reset "already sent today" and cause a duplicate (or a skip). |
| daemon claim path <-> catchup check path | Two code paths write/read the same sent_log key; if they key differently, a claim is orphaned or an alert is duplicated. |

Load-bearing test: `test_already_sent_slot_not_refired_after_tz_name_change`. |
