<objective>
Close the diagnosed UAT gap: the daemon's one-time startup "online" Discord ping
never reaches production. The `--run` CLI path calls
`daemon.run_daemon(config, settings, db_path)` WITHOUT `channel=`
(weatherbot/cli.py:480), so `run_daemon`'s `channel` defaults to `None`, is forwarded
as `None` into `emit_online`, and `emit_online`'s `if channel is not None:` guard
(daemon.py:527) silently drops the ping. Scheduled briefings are unaffected because
`fire_slot -> send_now` builds its own channel from config+settings (cli.py:119-122);
only the daemon's online ping has no such fallback. Confirmed live on host
`yahir-mint` after a real `sudo reboot`: service active, `weatherbot online jobs=3`
logged, READY=1 reached systemd, heartbeats ticking — but no Discord message arrived.

Fix it at the existing composition-root seam: in `run_daemon`, when `channel is None`
and `settings is not None`, build the channel ONCE via `build_channel(config, settings)`
and thread that single instance into BOTH `emit_online` AND `_register_jobs` — exactly
mirroring `send_now`'s fallback — while keeping `channel=` injectable for tests and
keeping `emit_online`'s best-effort guard intact. Then add a regression test that drives
the `channel=None` path and asserts the ping is sent, closing the test blind spot that
let this ship.

Purpose: Restore the third (human-facing) half of the startup online signal so a
silent death after deploy/reboot is detectable in the Discord channel, not just in the
journal (OPS-02 SC#3 / OPS-03 online-signal intent).
Output: A 2-line-ish fallback in `run_daemon` + a regression test; all 184 existing
tests stay green.
</objective>

  <name>Task 1: Build the channel from settings in run_daemon when channel is None</name>
  <files>weatherbot/scheduler/daemon.py</files>
  <action>
In `run_daemon` (daemon.py ~536), add a fallback that constructs the delivery channel
when none was injected, mirroring `send_now`'s seam at cli.py:119-122. Place it at the
TOP of the function body, BEFORE the `_register_jobs(...)` call (daemon.py ~558) so the
single built instance can be threaded into both the jobs and `emit_online`:

  - If `channel is None` and `settings is not None`: call `build_channel(config, settings)`
    once and reassign the local `channel`. Use the existing factory:
    `from weatherbot.channels import build_channel` (the package re-exports it; confirm
    via `weatherbot/channels/factory.py::build_channel`). Prefer a LAZY in-function
    import (inside `run_daemon`) to stay consistent with the module's existing lazy
    imports (`from weatherbot.cli import send_now` is imported in-function at daemon.py:165
    to avoid the cli<->daemon cycle) and to keep `build_channel`'s transitive imports off
    the daemon module's import-time graph.
  - If `channel is None` and `settings is None`: leave `channel` as None (do NOT raise).
    This preserves the existing channel-less call path — `emit_online`'s
    `if channel is not None` guard (daemon.py:527) and the jobs' own None-handling
    (fire_slot -> send_now builds from settings) already tolerate None, and a test or
    future caller may legitimately drive the daemon with neither.

After this fallback runs, the EXISTING `_register_jobs(..., channel=channel, ...)`
(daemon.py:564) and `emit_online(..., channel=channel, ...)` (daemon.py:625) calls
automatically receive the built instance — do NOT add a second build site; the single
reassignment at the top is the only construction point (one channel shared by the online
ping and every briefing job, matching send_now's single-construction posture, WR-04).

Do NOT modify `emit_online`: keep its `if channel is not None:` guard AND its non-ok
`"online ping not delivered"` warning exactly as-is (daemon.py:527-533) — a genuinely
channel-less call must still not crash.

build_channel can raise ValueError on an unknown channel type or (downstream) a missing
webhook URL. Make a DELIBERATE, documented choice consistent with the project's
fail-loud-at-load posture: do NOT wrap the build in a try/except that would let the
daemon start ringless or crash-loop silently. Let a build_channel ValueError propagate
out of run_daemon — it raises at startup BEFORE the self-check gate and BEFORE
scheduler.start(), so a misconfigured channel surfaces immediately as a loud startup
failure (the same way a bad config fails loud at load) rather than a daemon that comes
"online" with no delivery path. Add a one-line comment at the fallback explaining this:
the build is intentionally un-guarded so a missing/invalid webhook is a real, visible
config error, not a silently-degraded online daemon. (Note: a transient OpenWeather
outage does NOT reach here — that is the self-check gate's job; build_channel only
assembles the Discord webhook client and does not make a network call.)

Update the `run_daemon` docstring with one sentence: when `channel is None` and
`settings` is present, the daemon builds the channel from config+settings (mirroring
send_now) and shares that single instance across the online ping and all briefing jobs;
an explicitly-injected channel wins and skips the build.
