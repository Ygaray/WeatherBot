<objective>
Create the read-only fetch→render core in a NEW `weatherbot/interactive/lookup.py`:
`lookup_weather()` resolves a configured location, fetches imperial+metric via the
existing One Call client, builds a `Forecast`, renders the exact v1 template, and
returns a `LookupResult(text, forecast, location)`. It writes NOTHING to the store
(D-06, criterion #2) and raises a typed `UnknownLocationError(ValueError)` on an
unknown name (D-07). The `resolve_location` raise is upgraded in place so the whole
v1.0 path inherits the richer error backward-compatibly.

Purpose: One read-only core, two future surfaces (P7 CLI prints `.text`; P11 Discord builds an embed from `.forecast` without re-fetching). This is the seam send_now will delegate to in plan 06-03.
Output: `weatherbot/interactive/lookup.py` (`lookup_weather`, `LookupResult`, `UnknownLocationError`), the `resolve_location` raise-upgrade in `weatherbot/config/loader.py`, and `tests/test_lookup.py`.

NOTE: This plan does NOT create `weatherbot/interactive/__init__.py` — that barrel is owned
exclusively by plan 06-03 so 06-01 and 06-02 run in parallel without a shared-file conflict.
Until then, `tests/test_lookup.py` imports directly from the submodule
(`from weatherbot.interactive.lookup import ...`); `pythonpath=["."]` makes the directory
importable without an `__init__.py`.

LOCKED DECISIONS adopted here: D-05 (LookupResult value object), D-06 (zero store writes),
D-07 (UnknownLocationError subclasses ValueError, carries requested + valid_names — and we
adopt RESEARCH Open Question 3 recommendation: upgrade resolve_location itself), Open
Question 1 recommendation (lookup_weather computes its own location-local sent_at/checked_at),
Pitfall 3 (lazy-import build_client inside the `client is None` branch to avoid the
cli↔interactive cycle).
</objective>
