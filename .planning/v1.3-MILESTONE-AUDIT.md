## Milestone Goal (recap)

Make the bot **tap-to-drive** — a pinned, always-live Discord control panel (location dropdown + command-button grid, results rendering in-place) becomes the operator's primary, typing-free way to access every existing read-only command. ## 2. Cross-Phase Integration

Integration checker (gsd-integration-checker) result: **0 orphaned exports, 0 missing connections, 0 broken flows, 0 unprotected routes.** Every v1.3 requirement participates in the shared dispatch / render / persistence spine. Highlights:

- **Single dispatch path (PANEL-10):** the arg-adaptation ladder lives in exactly one place — `dispatch_reply` in `interactive/dispatch.py`. Three callers route through it: bot `on_message`, CLI, and panel. `grep` for `spec.handler(` across `weatherbot/` returns only `dispatch.py`. No parallel command list.
- **Forecast via the additive seam (PANEL-07):** `panel.py` has no direct `cache.lookup` / forecast handler / `parse_forecast_flags` / `run_in_executor` — forecast sub-buttons build `ForecastFlags` and pass `flags=` through the same `dispatch_spec`. One forecast path shared with bot + CLI.
- **Persistence covers all children (PANEL-09):** `setup_hook` `add_view(PanelView(...))` registers all 13 static `custom_id`s built in `__init__`, including the Phase-19 toggle + 4 sub-buttons; reveal/collapse is a cosmetic `_render_view` clone that never drops children, so revealed sub-buttons still route post-restart.
- **Clone trap closed (PANEL-12/13):** the load-bearing `_render_view` clone carries `emoji=child.emoji` and re-derives the dropdown `default=` from `_selected_location` (never read back from `Select.values`), proven by dedicated clone-path tests.
- **Failure isolation (PANEL-11):** panel runs on `BotThread`'s own loop with a frozen read-only `DaemonState`; `scheduler/` imports neither `panel` nor `dispatch`; per-callback non-propagating envelope + `View.on_error` backstop. Re-proven against a **live `BackgroundScheduler`** with an `await`-shaped hanging callback — the sentinel briefing still fires.

### Found + fixed during live UAT (a real milestone blocker the unit suite missed)

- **panel-dead-after-first-tap** (`/gsd-debug`, fix `b48abc6`): `_render_view` rebuilt the
  message-bound clone from **plain** `discord.ui.Button`/`Select` whose callback is a no-op in
  discord.py 2.7.1; discord.py routes component taps by `message_id` first, so every tap after the
  first hit dead callbacks → "This interaction failed" with no server log. Fixed by rebuilding the
  clone from the real callback-bearing subclasses via `_clone_child`; two clone-routing regression
  tests (command + dropdown) added (RED→GREEN). The 649-test suite missed it because tests called
  `panel.on_select/on_command` directly, never routing a *second* tap through the rendered clone —
  exactly the gap Gate-2 exists to catch.

### Product-owner UX changes accepted during UAT

- **Always-visible 2×2 forecast grid** (quick task **260626-u8y**, `394162b`): removed the Forecast
  expand toggle + `_expanded` state machine; the four variants are now permanently shown (rows 3–4,
  fills Discord's 5-row max). **Supersedes PANEL-07's** original "tap to reveal" two-tier design
  (REQUIREMENTS.md + ROADMAP annotated). A forecast-button clone-routing regression test was added.
- **`!panel` re-summons to channel bottom** (quick task **260626-uqp**, `d83aa27`): flipped PANEL-01
  from idempotent reuse-in-place to create-fresh-at-bottom + pin + delete old (mobile findability —
  the pinned panel scrolls away on mobile). Create-before-delete preserves the exactly-one/no-orphan
  invariant; all Phase-18 prohibitions (perm preflight, Forbidden backstop, marker ownership,
  delete-not-unpin, operator gate) preserved.
